一、先建立心智模型：两条轨道

持久化轨道 · 改 conversation.jsonl

会话压缩（Compaction）

默认关闭（contextCompaction.enabled=false）。开启后才会主动 truncate / re-evict / LLM 摘要。API 报 context 超长时仍会强制 compact 一次。

发送轨道 · 不改磁盘

发送边界卫生（Hygiene）

默认开启。每轮调用模型前，把即将发出的 AgentModelMessage 瘦身；磁盘上的 tool 结果仍是完整的。Footer 显示 saved ~XK (hygiene)。

层级 改会话文件？ 何时触发 默认阈值（源码）
① Tool result eviction 是 单次 tool 返回后 结果 > 80,000 字符 → 写 evicted/*.txt，会话留 preview 2,000 字符
② truncateArgs 是 主动压缩开启后 ≥ 25 条消息 或 ≥ 40k 估算 token；旧 assistant 的 tool 参数截到 2,000 字符
③ prefix re-evict 是 动态模式 High+；静态达 truncate 阈值时 进一步收紧前缀里已有的 [Tool result evicted…] placeholder
④ LLM conversation compact 是 静态 ≥50 条/80k token；动态 ≥80% 窗口 前缀送摘要模型 → 替换为 __compaction_summary__ + 保留尾部 20 条
⑤ RequestHistoryHygiene 否 每轮 API 前（含 overflow 重试） tool 结果 > 32KB / 320 行 / 8k token → 发 API 时省略大部分行
⑥ ToolStorm 是（写 suppressed 结果） 同 turn 内重复 tool 相同 name+args 第 3 次起拦截（threshold=3）

场景 A：静态压缩（dynamicCompaction.enabled = false，compaction.enabled = true）

会话已累积 52 条消息，估算历史 82,000 token（超过 compact 阈值 50 条 / 80k）。

Step 1 · truncateArgs（若已达 25 条/40k）

前缀 assistant 消息里 file_edit 的 content 参数从 15,000 字符 → 2,000 字符 + "...(argument truncated)"

Step 2 · prefix re-evict（v1.1.22+ 静态路径）

达 truncate 阈值时，前缀 evicted placeholder 的 head/tail 再收紧（previewChars/2）。

Step 3 · LLM 摘要 compact

前缀 32 条 → 摘要 LLM；保留最近 20 条 tail。

场景 B：动态压缩（dynamicCompaction.enabled = true）

假设 contextWindowTokens=65535，固定开销（system+tools）≈ 12,000 token，可用历史预算 ≈ 53k。当前历史估算 42k → 利用率 ≈ 64%（Normal，不压）。

利用率压力级别动作
< ~55%Normal无（除非静态 floor 已达）
≥ 72%（80%×0.9）HightruncateArgs + prefix re-evict
≥ 80%Critical上述 + LLM compact → 压到 ~30% 窗口
API overflowOverflow强制 compact → ~20% 窗口后重试

数值推演：历史从 42k 涨到 46k（≈72%）

① truncate 后
~38,000 tok
② re-evict 后
~33,000 tok

继续涨到 48k（≥80% Critical）→ LLM compact

目标保留
~30% × 65k ≈ 16k hist

动态模式用 PromptPressureStore：若 API 上一轮回报 prompt_tokens=52000 而估算只有 46000，预算按真实值校正，更早触发压缩。

场景 E：Context 超长 → Overflow 重试

  1. 第一次 CompleteAsync 抛 context_length exceeded
  2. PreCompletionPipeline(ForceCompact, Overflow) — LLM 摘要，目标 ~20% 利用率
  3. modelMessageCache.Invalidate()
  4. ModelMessagesForApiBuilder.Build — 必须经 Hygiene（v1.1.22 修复点）
  5. 第二次 API 调用；footer 累计 hygiene savings

数据变化示意

Attempt 1:  prompt ≈ 68,000 tok  →  API Error: context_length
Compact:     120 msgs → 22 msgs,  TokensBefore 61000 → TokensAfter 14000
Attempt 2:   hygiene 再减 tool 尾部 →  prompt ≈ 13,500 tok  →  Success
